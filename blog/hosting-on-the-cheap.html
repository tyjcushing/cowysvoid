Hosting a Custom Website on the Cheap

How I built Cowy’s Void with Cloudflare Pages, GitHub, and VS Code.

webdev cloudflare github vs code 6 min read Oct 8, 2025
One of my favorite things about the modern web is that you don’t need a huge budget or enterprise tools to launch your own site. Cowy’s Void runs almost entirely for free using Cloudflare Pages, GitHub, and VS Code. Here’s exactly how I set it up and the small tricks that made it smooth.

Why this stack?

  • Free static hosting Cloudflare Pages
  • Easily deploys push to main → live
  • Simple Security Managed by Cloudflare SSL/CDN/caching handled
  • New Dev-friendly VS Code + GitHub workflow

Step 1 — Authoring in VS Code

I wrote this site in plain HTML/CSS/JS inside VS Code. No framework. I looked for inspiration on dark cosmic palette, rounded cards, etc. Helpful extensions for VS code: Prettier (formatting), Live Server (quick preview), and the built-in Git panel.

Step 2 — Version control with GitHub

I created a repo (e.g. cowys-void) and keep main clean. Experiments live on feature branches so prod never breaks. 

git init
git remote add origin git@github.com:<you>/cowys-void.git
git add .
git commit -m "Initial site"
git push -u origin main

Step 3 — Hosting with Cloudflare Pages

  1. In Cloudflare Pages → Create Project
  2. Connect your GitHub repo and choose the deploy branch (main for me).
  3. Build settings: for a plain static site, no build command needed; output directory is the repo root.
  4. Click Deploy — you’ll get a Cloudflare URL immediately.

Every push to main redeploys the site. PRs get preview URLs, which is great for testing.

Step 4 — Custom domain + SSL

I registered a domain with Cloudflare (you can also bring your own). In Pages → Custom domains, I mapped it to the project. SSL is automatic, and DNS is handled for me.

Step 5 — Partials & background video

To keep pages small and consistent, I load /partials/nav.html, /partials/footer.html, and a Discord widget at runtime. I also conditionally load a background video only for fast connections. You’ll see the same script across my pages.

Gotchas I hit (so you don’t)

  • Cache vs. updates: add { cache: 'no-store' } to fetch for partials/data so content updates show up immediately.
  • Relative paths: keep asset and partial paths absolute (e.g. /partials/nav.html) to avoid subfolder surprises.
  • Preview vs prod: Cloudflare preview URLs are a different origin; avoid hard-coding the prod domain in scripts.
